# 博闻 — 代码书写规范

本文约束后续开发风格，目标是让 MVP 快速演进但不变成临时脚本堆。

## 1. TypeScript 规则

- 新代码使用 TypeScript。
- 公开函数参数和返回值显式标注类型。
- 平台原始响应单独定义 interface，并在 infrastructure 内映射成 domain 类型。
- 避免 `any`；确实需要时限制在 adapter 边界。

## 2. 函数设计

- 纯业务逻辑写成纯函数。
- 函数名使用动词短语：`rankHotVideos`, `mapBilibiliItem`, `generatePlan`。
- 排名、过滤、分析规则不要直接读取环境变量。
- 单个函数超过 80 行时优先拆分。

## 3. 错误处理

- 不在深层函数里吞错误，除非返回明确 fallback 结果。
- catch 后至少保留错误 message。
- API 返回用户能理解的文案，日志保留技术细节。

## 4. 注释

需要注释的场景：

- 排名公式。
- 增长阈值。
- fallback 原因。
- 平台字段映射中特殊处理。

不需要注释：

- 变量名已经表达清楚的赋值。
- 重复代码本身含义。

## 5. UI 规则

- 品类切换、Top10、视频链接、分析结果是核心工作流，不能被装饰性布局遮挡。
- 数据来源 `live/fallback` 必须可见。
- 长标题、长建议必须在移动端不溢出。
- 组件变复杂后拆到 `src/interface/components`。

## 6. 数据规则

- 播放量等指标用 number。
- 时间用 ISO string。
- URL 用完整链接。
- 中文品类集中定义，不在 UI/API 中散落硬编码。

## 7. 提交前检查

- 类型无明显错误。
- `npm test` 通过。
- 涉及 UI 时本地打开检查品类切换。
- 涉及抓取时确认 fallback 仍可用。

一句话：真实世界的不稳定留在 adapter，博闻自己的业务代码保持清楚、可测、可替换。
